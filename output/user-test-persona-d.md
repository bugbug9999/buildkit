## 시나리오 1: 첫 진입 — 아내가 설치해준 앱을 처음 열어봄

### 실제 UI 구조 (코드 기반)

약이 등록되지 않으면 EmptyState 렌더링:
- 헤더: "투약 관리" + 우측 "약 관리" 링크
- 중앙: 💊 이모지 + "복용 중인 약을 등록해 주세요" + 설명 + [약 등록하기] 버튼

### 박정호의 행동 및 반응

**동작:** 아내가 "이거 설치했어"라며 건네줌 → 앱을 열어봄 → 약이 없으니 빈 화면 → 중앙 카드의 초록 [약 등록하기] 버튼이 눈에 들어옴 → 바로 누름

**감정:** "오, 뭐하는 앱이네. 약 등록하라고 하네"라는 느낌으로 시작. CTA가 명확해서 혼란 없음.

### 잘 된 부분

- 💊 이모지와 큰 버튼으로 "약 관련 앱"이라는 것을 즉시 이해
- "복용 중인 약을 등록해 주세요"라는 문구가 직관적
- 초록 버튼이 충분히 크고 눈에 띔

### 혼란 포인트

- **우측 상단의 작은 "약 관리" 텍스트 링크** — fontSize 14, 초록 텍스트만. 박정호는 이것이 버튼인지 몰 수 있음. [약 등록하기] 버튼과 같은 목적인지 다른 목적인지 모호. "둘 중 뭘 눌러야 하지?" 혼란 가능.
- "오늘 복약 체크, 달력 기록, 주간 요약을 한 번에 관리합니다"라는 설명만으로는 "이게 내게 왜 필요한가"를 이해하기 어려움. "약 빼먹지 않도록 도와줌" 같은 더 직관적인 언어가 필요.

## 시나리오 2: 약 3개 등록 — 리벨서스 + 메트포르민 + 혈압약

### 약 등록 화면 구조 (AddScreen 코드 기반)

필드 순서:
1. 약 이름 (TextInput)
2. 복용 형태 — 경구 / 주사 / 기타 (Segment)
3. 용량 (TextInput)
4. 복용 주기 — 매일 / 요일 지정
5. 복용 시간대 — 바텀시트로 추가
6. 복용 조건 — 공복 / 식전 / 식후 / 무관
7. 메모 (multiline TextInput)
8. 우상단 [저장] 버튼

### 2-1. 리벨서스 등록 (14mg, 아침 공복)

**동작:**
"약 이름" → "리벨서스" 입력 → 형태 "💉 주사" 선택 → 용량 "14mg" 입력 → 주기 "매일" → [+ 시간대 추가] → 바텀시트에서 "아침 08:00" 선택 → 복용 조건 "공복" 선택 → 저장

**박정호의 심리:**
- "리벨서스"는 일반명칭 모르고 약국에서 받은 명칭 그대로 입력
- "주사"를 선택할 때 "아, 피하주사 바늘 맞는 거구나" 확인
- 아침 공복이라는 것을 처방약 복용 습관으로 알고 있어서 자동으로 고르는 편

**혼란 포인트:**
- **"복용 조건" 용어의 혼란** — "공복", "식전", "식후", "무관"이라는 용어가 약국 영수증에서도 봤지만 정확한 의미를 완전히 이해하지 못함. 특히 "식후"와 "무관"의 차이를 "그럼 뭘 고르지?" 하며 고민. 설명 없음.
- **우상단 [저장] 버튼이 텍스트만** — 약국 앱이나 카카오톡은 하단에 큰 버튼 있는데, 이걸 상단에만 있어서 "이게 저장이구나"를 늦게 깨달음
- 바텀시트에서 "아침 08:00" 빠른 선택이 너무 당연하게 작동해서 좋긴 한데, "시간을 커스텀할 수도 있겠구나"는 발견 안 함. 기본값으로 만족.

**잘 된 부분:**
- "13mg" vs "14mg" 입력할 때 폰트가 커서(fontSize 15 + bold) 정확히 입력 가능
- 주사/경구 구분이 아이콘(💉 vs 💊)으로 되어 있어서 직관적

### 2-2. 메트포르민 등록 (500mg, 아침+저녁 식후)

**동작:**
약 이름 "메트포르민" → 경구 → "500mg" → 매일 → [+ 시간대 추가] → "아침 08:00" 선택 → 다시 [+ 시간대 추가] → "저녁 19:00" 선택 → 식후 → 저장

**박정호의 심리:**
- 리벨서스는 주사니까 쉬웠는데, 이번엔 약이름이 "메트포르민"이라는 영어명칭이라 약간 낯섦. 하지만 처방약이니까 입력함.
- 아침/저녁 두 번 먹는다는 것을 알고 있어서 [+ 시간대 추가]를 두 번 누르는 흐름을 (어렵긴 하지만) 발견. 두 번째 추가할 때 "이미 아침이 추가된 상태에서 저녁만 선택 가능"한 바텀시트가 다시 열리는데 혼란.

**혼란 포인트 — 이 부분이 가장 심각:**
- **[+ 시간대 추가] 버튼이 작고 눈에 안 띔** — fontSize 14, 테두리만 있는 흰 배경. 아침을 추가한 후 버튼을 다시 찾는 게 어려움. 카드가 추가되고 나면 버튼이 올라가면서 위치가 변하기도 함.
- **"저녁 19:00"을 선택해도 실제 복용 시간이 19시인지 확인 불가** — 식당 일정상 저녁 약을 19시가 아니라 20시에 먹을 수도 있는데, 고정값 19:00으로 설정되면 미복용 배지가 잘못 뜰 수 있음. 직접 입력 기능은 있지만 바텀시트에서 이 기능을 찾기 어려움.
- 코드상 `draft.schedules = [{ time: "08:00" }, { time: "19:00" }]`로 추가되지만, 박정호는 이미 추가된 것을 시각적으로 확인하고 싶음. 바텀시트를 닫으면 메인 폼에 "08:00, 19:00" 같은 요약이 있으면 좋은데 현재는 카드 형태로 쌓임.

**잘 된 부분:**
- 시간대를 여러 개 추가할 수 있는 구조는 정확히 기획서를 충족하고 있음
- 이미 추가된 시간대는 바텀시트에서 비활성화되어 중복 방지됨

### 2-3. 혈압약 등록 (5mg, 아침 매일)

**동작:**
약 이름 "혈압약" → 경구 → "5mg" → 매일 → 아침 → 무관 → 저장

**박정호의 심리:**
- 혈압약은 일반인이 이해하기 쉬운 용어
- 복용 조건이 "무관"인데 이 단어가 헷갈림. "아무 때나 먹어도 된다"는 뜻인지 확실하지 않음. 하지만 "혈압약은 언제 먹어도 괜찮다"는 의료 상식으로 "무관"을 고름.
- 저장 완료 후 메인으로 돌아감

**혼란 포인트:**
- **3개 약 등록에 필요한 단계 수** — 첫 약은 등록 화면 진입 + 약 이름~메모 + 저장 = 약 8~9단계, 두 번째도 약 9단계(두 번 저장까지 포함). 총 최소 15단계 이상 필요. 박정호는 식당 일로 바빠서 "이거 왜 이리 길어?" 하는 생각 들 수 있음.
- 저장 후 바로 메인으로 이동하는데, 연속으로 다른 약을 등록하려면 다시 [약 관리] → [+ 새 약 등록하기]를 눌러야 함. 기획서에는 "다른 약도 등록할까요?" 프롬프트가 있어야 하지만 미구현.

**전체 등록 요약:**
- 3개 약 모두 성공적으로 등록됨
- 메트포르민 시간대 이중 등록이 가장 어려웠던 부분
- "저장" 버튼 위치 개선 필요

## 시나리오 3: 식단 기록 시도 — 식당 밥을 입력해서 칼로리 추적하고 싶은 욕구

### 앱의 기능 검토

박정호는 "GLP-1 약물을 먹으니까 음식도 관리해야 한다"는 생각에 앱에서 식단을 기록할 수 있을 거라고 기대. 메인 화면을 이리저리 탭해보며 찾음.

### 현실 (기획서 기반 범위)

기획서에는:
- 7.5절 "식단 기록 화면"
- 칼로리 + 음식명 + 시간 입력

하지만 injection.tsx 코드에 식단 관련 스크린이 없음. 현재는 "투약 관리"만 있고 식단은 별도 탭이거나 미구현 상태.

### 박정호의 반응

**동작:**
메인 화면 스크롤, 탭 바 다른 버튼들 찾기, "어디서 밥 입력하지?" 고민

**감정:**
- 초반: "오, 약도 기록하고 음식도 기록할 수 있는 앱이겠네"라는 기대
- 중반: "어디서 식단을 입력하지?" 하며 헤매기 시작
- 후반: "아, 이 앱은 약만 기록하는 거구나" 깨달음

### 혼란 포인트 및 아쉬움

- **기획서에는 식단 기능이 있지만 현재 코드에는 없음** — 박정호의 기대가 충족되지 않음. "약만 관리하는 앱이면 그냥 종이에 기록하는 것과 뭐가 다르지?"라는 회의감 생김.
- 병원에서 "식사와 함께 약 먹으세요"라는 설명을 받았어서 식단 기록의 필요성을 느끼는데, 앱이 지원 안 함.
- 다른 약 관리 앱(예: MyTherapy, Medisafe)을 사용해본 적은 없지만, "좋은 앱이면 약이랑 음식까지 한 번에 관리할 수 있지 않을까?" 예상

**NPS 판단에 영향:**
이 시점에서 "앱이 유용하지 않을 수도 있겠다"는 의구심이 살짝 생김. 하지만 약 기록 기능은 충분하므로 계속 사용할 가능성은 있음.

## 시나리오 4: 오늘 아침 복약 체크 — 세 약을 한 번에 챙김

### 실제 UI 구조 (MainScreen 코드 기반)

등록된 약이 있으면:
- "오늘의 복약" N/M 완료 배지 + 진행률 바
- 시간대별 그룹: 아침(리벨서스, 메트포르민, 혈압약 3개) / 저녁(메트포르민 1개)
- CheckCard: 이름 + 메타정보(용량·조건·시간) + 우측 체크 버튼
- 달력 섹션 (스크롤 하단)
- 주간 요약

### 박정호의 행동

**동작:**
아침 일찍 일어나서(6시) 약을 먹기 전 앱을 열어봄 → "오늘의 복약" 섹션 확인 → 아침 그룹에 리벨서스, 메트포르민, 혈압약 3개 카드가 보임 → 우측 원형 체크 버튼을 3개 다 누름 → "3/3 완료"

**감정:**
"오, 이게 지금 뭐가 남았는지 보여주는 거네. 편하네." → "체크하면 카드가 옅은 초록으로 바뀌니까 다 됐다는 게 확 와닿네"

### 잘 된 부분

- 시간대별 그룹(아침/저녁)으로 박정호가 아침에 필요한 약들이 한눈에 보임
- 진행률 배지("3/3 완료")와 바가 시각적으로 만족감 제공
- 체크 후 카드 배경이 초록으로 바뀌는 피드백이 명확
- 약 이름 글씨가 충분히 커서(fontSize 15, bold) 읽기 쉬움

### 혼란 포인트 — 접근성 이슈

- **체크 버튼이 작음 — 노안 + 정밀도 문제** 
  - CheckButton width:32, height:32 (권장은 44pt)
  - 박정호가 노안 초기에 들어섰어서 버튼이 작으면 정확히 누르기 어려움
  - "어, 안 눌렸네?" 하고 두 번 누르거나 옆을 잘못 누를 수 있음
  - 특히 아침에 피곤한 상태에서 조작하면 실수율 높음

- **체크 버튼만 탭 가능 — 카드 전체 탭 불가**
  - 기획서: "약물 카드를 탭하면 체크" 명시
  - 현재 코드: 카드 내부의 TouchableOpacity 버튼만 탭 영역
  - 박정호 같은 일반 사용자는 "카드를 누르면 체크되겠지"라고 직관적으로 생각하지만 그렇지 않음
  - 버튼 위치를 정확히 찾아야 하는 불편함

- **복약 카드 메타 텍스트가 작음**
  - checkMeta fontSize:12, color: SUBTEXT (#94A3B8)
  - "14mg · 공복 · 08:00" 같은 정보가 12px로 표시
  - 노안 초기인 박정호가 정보를 읽으려면 화면에 가까이 가야 함
  - "용량이 뭐지? 아 14mg 맞네" 하는 확인 과정이 번거로움

- **어제 미복용 약에 대한 인앱 알림 없음**
  - 기획서: 다음 날 진입 시 "어제 N개 약을 놓쳤어요" 배너 표시
  - 현재 코드: 이런 알림 없음
  - 박정호는 어제 저녁 메트포르민을 빼먹었을 수도 있는데 앱이 알려주지 않음

## 시나리오 5: 저녁 복약 — 메트포르민 한 개 체크

### 행동

저녁 19시 경, 식당이 바쁜 와중에도 앱을 열어서 저녁 약을 체크하려고 함 → 메인 화면의 저녁 섹션으로 스크롤 → 메트포르민(저녁) 카드 찾기 → 체크

**감정:**
- "식당이 바쁘면 이 약을 놓치기 쉬운데, 앱이 있으니까라도 챙길 수 있겠네"
- "체크했으니까 기록이 남는 거겠지. 병원에서 이거 보여주면 좋겠다"

### 혼란 포인트

- **저녁 섹션을 찾기 위해 스크롤 필요**
  - 아침 섹션 다음에 저녁 섹션이 오는데, 화면이 작으면 스크롤 필요
  - 박정호가 바쁜 와중에 "저녁 약이 어디 있지?" 하며 찾는 과정 불편
  - 카드가 눈에 확 들어오는 것도 아니고 검색 기능도 없음

- **미복용 알림 없음**
  - 저녁 약을 빼먹어도 "아, 너 저녁 약 못 먹었지?" 알림이 없음
  - 기획서에는 복용 예정 시간 30분 경과 시 푸시 알림이라고 했지만 미구현
  - 박정호는 스스로 앱을 열어야만 빠뜨린 약을 인지 가능

---

## 시나리오 6: 저녁 약 빼먹음 — 식당 바빠서 메트포르민 놓침

### 상황 설정

어제 저녁에 메트포르민을 먹으려고 했지만 손님이 몰려서 까먹음. 오늘 아침에 앱을 열었을 때.

### 박정호의 기대 vs 현실

**박정호의 기대:**
"어제 안 먹은 약이 있으면 앱이 알려줄 거야"

**현실 (코드):**
- MainScreen 진입해도 어제 누락 약에 대한 알림 배너 없음
- 어제 저녁 메트포르민 빠뜨린 것을 인지하려면 달력 보러 스크롤 해야 함
- `getCheckItemsForDate(medications, logs, yesterday)` 로직이 없어서 자동 감지 불가

### 박정호의 행동

**시나리오 1: 긍정적 경로 (10% 확률)**
- 어제 저녁을 뺀 것을 기억하고 있었음
- 달력을 스크롤해서 어제 날짜 찾음
- 어제 날짜의 메트포르민 카드를 찾아 체크
- "어제 먹은 것처럼 기록해야 하나?" 고민하지만 어쨌든 체크함

**시나리오 2: 부정적 경로 (90% 확률)**
- 저녁 약을 뺀 것을 자명하게 인식하지 못함
- 앱에서 "어제 뭔가 빠뜨렸어요" 안내 없음
- "어, 뭔가 빠뜨린 게 있나?" 의심해도 앱이 도와주지 않음
- 결국 포기하고 "다음부터는 챙겨야지"하고 넘어감

### 혼란 포인트 — 이 부분이 앱의 치명적 약점

- **푸시 알림 미구현**
  - 기획서 명시: 복용 예정 시간 30분 경과 시 알림
  - 코드: 알림 코드 0줄
  - 박정호 같은 바쁜 사용자가 자발적으로 앱을 열지 않으면 빠뜨린 약을 모를 수밖에 없음

- **다음 날 인앱 알림 없음**
  - 기획서 명시: "어제 메트포르민(저녁)을 놓쳤어요" 배너
  - 코드: MainScreen 상단에 이런 로직 없음
  - 유저가 스스로 달력을 찾아야 함

- **달력 접근의 어려움**
  - 달력이 화면 하단에 있어서 스크롤을 많이 내려야 함
  - 달력 도트가 5px로 작아서 어제 미복용을 시각적으로 파악 어려움
  - "아, 어제가 일부 완료(노랑) 또는 미복용(빨강)이구나"를 인식하려면 달력을 자세히 봐야 함

- **달력 날짜 탭 후 바텀시트 흐름의 복잡성**
  - 어제 날짜를 탭하면 DateDetailSheet가 열림
  - 거기서 메트포르민(저녁) 카드를 찾아서 체크
  - 박정호는 이 흐름을 처음에 발견하지 못할 가능성 높음

### 종합 평가

이 시나리오는 **GLP-Care 앱의 가장 큰 약점**을 드러냄:
- 능동적 알림 없음 → 유저가 자발적으로 앱을 확인해야 함
- 빠뜨린 약을 발견하는 경로가 어려움 → 달력 스크롤 + 날짜 탭 + 바텀시트
- "식당 운영하느라 바쁜" 박정호 같은 유저에게 가장 필요한 기능이 없음

## 시나리오 7: 달력 확인 — 지난 주 복약 패턴 보기

### 실제 UI 구조 (CalendarView 코드 기반)

- 월 단위 달력 (이전/다음 월 이동)
- 날짜 셀: 날짜 숫자 + 배경색 원 + 하단 도트
- 상태별 색상: 완료=초록, 일부=노랑, 미복용=빨강
- 날짜 탭 → DateDetailSheet

### 박정호의 행동

메인 화면 스크롤 → 달력 섹션 도달(6~7번 스크롤) → 지난주 날짜들을 시각적으로 확인

### 감정

- "오, 이 주에 초록색이 많으니까 잘 챙겼구나"라는 만족감
- "노랑이나 빨강 날짜가 있으면 그 날 뭘 빼먹었는지 볼 수 있겠네"

### 잘 된 부분

- 날짜별 색상(초록/노랑/빨강)으로 한눈에 복약 이력 파악 가능
- 도트가 약별 색상으로 표시되어 어떤 약을 복용했는지 구분 가능 (이론상)
- 바텀시트에서 해당 날 체크/미체크 수정 가능

### 혼란 포인트 — 접근성 관점

- **달력이 메인 화면 하단에 묻혀 있음**
  - 메인 레이아웃 순서: 헤더 → 약 요약 → 오늘 복약 → 시간대별 카드 → 다음 예정 → 달력
  - 스크롤 6~7번 필요
  - 박정호가 "빨리 지난주 복약 현황을 보고 싶다"는 욕구에 달력 접근이 어려움
  - 병원 진료 예약 날 "이 주 얼마나 잘 챙겼는지 보여주고 싶은데" 달력이 금방 안 나타남

- **달력 도트가 너무 작음**
  - calendarDot width:5, height:5
  - 노안 초기인 박정호가 도트를 제대로 인식하기 어려움
  - "어, 이 날짜에 약이 있었나?" 하며 확인이 어려움

- **달력 날짜 셀도 작음**
  - calendarDayCircle 32×32 (권장 44pt)
  - 7칸을 화면 너비에 나누기 때문에 구조적 한계
  - 박정호가 정확히 원하는 날짜를 탭하기 어려움
  - 옆 날짜를 실수로 탭할 수 있음

- **색상 범례 없음**
  - 초록/노랑/빨강이 뭘 의미하는지 설명 없음
  - 박정호는 "초록은 다 먹은 거겠지"라고 유추하지만 확실하지 않음
  - "달력 범례" 같은 것이 달력 아래에 있으면 좋을 텐데 없음

- **달력 이전/다음 월 버튼이 불분명**
  - "‹" "›" 특수문자로 표시
  - fontSize 22, color SUBTEXT
  - 박정호는 이게 버튼인지 몰 수도 있음
  - 명확한 "◀ 이전 달" "다음 달 ▶" 라벨이 필요

## 시나리오 8: 프로필/설정 — 필요한가?

### 박정호의 기대

"내 약 정보, 복약 기록을 어떻게 관리할 수 있을까?"

### 현실 (코드 검토)

현재 injection.tsx에는 프로필/설정 탭이 명확하게 구현되어 있지 않음. 약 관리 화면과 메인 화면 정도만 있는 상태.

### 필요한 설정 항목 (박정호 관점에서)

- 약 알림 시간 설정 (현재는 고정 08:00, 19:00 등)
- 복약 기록 내보내기 (병원에 보여주기 위해 PDF/이미지 형태)
- 과거 기록 조회 (3개월 전까지 어떻게 챙겼는지)
- 약 수정 기능 (용량 변경 시)

### 현황

- 약 삭제만 가능, 수정 불가 → 용량이 바뀌면 삭제 후 재등록 필요
- 기록 내보내기 기능 없음 → 병원에서 "약 복약 기록 있으세요?"라고 물으면 못 보여줌

### 박정호의 반응

"설정 화면이 있으면 좋겠는데 뭐가 있는지 모르겠네"라는 느낌으로 포기. 기본 체크와 달력 보기만 하게 됨.

---

## 시나리오 9: 앱 종료 후 재진입 — 데이터 유지 확인

### 박정호의 행동

앱을 종료 후 1시간 뒤 다시 열어봄 → 어제 등록한 3개 약과 오늘 아침 체크 기록이 있는지 확인

### 현실 (코드)

injection.tsx는 로컬 state(`useState`)만 사용:
- `const [medications, setMedications] = useState([])`
- `const [logs, setLogs] = useState([])`

앱이 재시작되면 초기화됨. AsyncStorage나 API 연동 없음.

### 박정호의 경험

**실제 결과:** 앱을 재시작하면 모든 약과 복약 기록이 사라짐

**박정호의 반응:**
"어? 아까 등록한 약이 다 사라졌네? 데이터가 저장이 안 되나? 이 앱 쓸 수 없겠는데?"

**NPS 영향:**
이 순간이 **앱 이탈의 결정적 순간**. 박정호는 "이 앱은 신뢰할 수 없다"는 판단을 내림.

### 평가

- **MVP 단계의 완전한 결함** — 데이터 영속성 없음
- 박정호 같은 일반 사용자는 "왜 저장이 안 돼?" 하며 다른 앱으로 이탈

---

## 시나리오 10: 결제 의향 — 유료화/광고 대응

### 박정호의 앱 결제 경험

- 앱 결제 거의 경험 없음 (카카오톡, 네이버는 무료)
- 앱 내 결제나 구독료 개념이 낯섦
- "스마트폰 앱은 기본 무료인 줄 알았는데?"

### 시나리오 1: 무료 버전에 광고 포함

만약 앱이 무료 대신 광고를 붙인다면:

**반응:**
- "광고? 약 복용하는 앱인데 광고까지?" 극도로 부정적
- 광고 클릭 실수로 다른 페이지 이동 가능성
- "이런 건 병원에서 받는 복약수첩이 낫네" 하며 이탈

**NPS 영향:** -5 (매우 부정적)

### 시나리오 2: 유료 구독 (월 2,900원 등)

**박정호의 심리:**
- "약 등록하는 앱에 돈을 왜 내?" 거부감
- 아내가 깔아줬으니 아내에게 "이거 돈 내야 해?"라고 물을 가능성
- 아내: "몇 천 원 정도면 돼, 약 빼먹지 말고 써"라고 조언할 수도 있음

**구매 의향:**
- 박정호 단독으로는 결제 안 할 확률 80%
- 아내 강권 시 "그럼 한 달만 해봐"라며 결제 가능성 20%

### 시나리오 3: 기본 무료, 프리미엄 결제 옵션

예: "기본 복약 기록은 무료, 병원 기록 내보내기는 프리미엄"

**박정호의 반응:**
- "기본이 무료면 일단 써볼 만하네"
- 실제로 기본 기능만으로도 충분하면 결제 안 함
- 병원에서 "기록 보고 싶으신데?" 라는 말 들을 때 "이거 돈 내야 보여줄 수 있어?"라며 결제 고려 가능

**NPS 판단:**
- 유료화가 아니면 계속 무료로 쓸 가능성 높음
- 광고는 절대 안 받음 (극도로 싫어함)

---

## 결제 vs 아내의 역할

### 아내의 개입

아내가 앱을 설치해줬다는 것이 중요:
- 아내가 "약 빼먹지 말라"고 챙기는 입장
- 결제가 필요하면 아내가 대신 할 가능성 있음
- 박정호는 "아내가 챙겨주는 느낌"에 만족하는 편

### 가족 공유 기능에 대한 욕구

만약 기획서에 "가족 공유" 기능이 있다면:
- 아내도 박정호의 복약 기록을 볼 수 있음
- "정호가 약 잘 챙기고 있나?" 아내가 확인 가능
- 이 기능이 있으면 아내가 "이 앱 좋네" 해서 결제 권유할 가능성 높음

---

## 병원 진료 기록 출력/공유

### 박정호의 기대

"내과 가기 전에 앱에서 '이 주 복약률 85%, 빠뜨린 약 2회'같은 요약을 보여주고 싶어"

### 현실 (코드)

- 주간 요약은 있음 (복약률 % + 미복용 횟수)
- 하지만 기록을 출력/공유/내보내기 기능이 없음
- 병원 가서 "약을 잘 챙겼어요?"라고 물을 때 "앱에 기록돼 있어요"라고만 말할 수 밖에 없음

### 아쉬움

병원에 가기 전날 "아, 앱에 기록을 스크린샷해서 가져가야 하나?"라는 생각 들 수 있음. 앱 자체에서 "지난 한 달 복약 기록" PDF를 생성해서 이메일로 보내는 기능이 있으면 좋을 텐데.

---

## NPS 점수 및 이탈 예측

### NPS 점수 계산 (0~10)

박정호가 이 앱을 친구/가족에게 추천할 의향도를 0~10으로 평가:

**긍정 요소:**
- 간단한 UI로 약 등록 가능 (체크 기능 직관적)
- 달력으로 복약 이력 시각화
- 아내와 함께 쓸 수 있는 느낌

**부정 요소:**
- 데이터 영속성 없음 (재시작하면 초기화)
- 푸시 알림 없음 (바쁠 때 빠뜨린 약 감지 불가)
- 저녁 약을 빼먹기 쉬운데 앱이 적극 도와주지 않음
- 식단 기록 기능 없음 (기대했던 기능)
- 달력 접근성 낮음 (스크롤 많이 필요)
- 복약 기록 내보내기 불가 (병원에서 보여줄 수 없음)
- 글씨 작음 (노안 초기에 힘든 부분)

**NPS = 4/10**

**이유:**
- 기본 복약 기록 기능은 있지만 중요한 기능(알림, 데이터 저장)이 빠짐
- "이 앱 써봐?"라고 물으면 "음, 약 빼먹지 말고 싶으면 써볼 만하지만 아직 부족한 게 많네"
- 친구에게 자신 있게 추천하기는 어려움

## 3개월 후 이탈 예측

### 시나리오 1: 앱 재시작 시 데이터가 살아있다면 (개선 시)

- 아내가 "계속 써, 약 잘 챙기려면"이라고 권유
- 매일 아침 습관처럼 앱 열어서 체크
- **계속 사용 확률: 70%**

### 시나리오 2: 현재 상태 그대로라면 (개선 없음)

- 첫 주: "좋은데 자꾸 데이터가 없어지네" 불만
- 2주: "그냥 아내 말 들으면서 손수 챙기자"라며 포기
- 3주: 앱 삭제
- **이탈 확률: 85%**

## 우선 개선 요청 3가지 (박정호 목소리)

### 1순위: "데이터가 자동으로 저장되게 해주세요"

> "아이고, 앱을 닫았다 열면 약이 다 사라져? 이게 뭐하는 앱이야. 차라리 종이에 쓰지. 꼭 약국 영수증처럼 데이터가 남아야 다음에 또 쓸 수 있지."

**개선:** AsyncStorage 또는 서버 API로 데이터 영속성 확보

### 2순위: "저녁 약을 빼먹으면 알려줘요"

> "식당이 바쁘면 저녁 약을 자꾸 빼먹어. 앱이 '야, 저녁 약 먹을 시간이다'고라도 알려줘야지. 지금은 내가 일일이 들어가서 봐야 하니까 번거롭고, 또 깜박하면 아무 소식이 없으니까 약을 빼먹은 줄도 몰라."

**개선:** 푸시 알림 구현 (복용 예정 시간 30분 경과 시)

### 3순위: "글씨를 더 크게 해주세요"

> "약 이름은 괜찮은데, 용량이랑 시간이 너무 작아서 안경 벗고 봐도 잘 안 보여. 이 정도면 그냥 나이가 들고 있다는 신호인데 (웃음), 앱이라도 조금 크게 해줄 수 있지 않나. 특히 약의 용량은 중요하니까."

**개선:** checkMeta fontSize 12 → 14 이상

## 종합 UX 평가

### 잘 구현된 부분

1. **Empty State → 약 등록 → 체크 플로우가 논리적**
   - 처음 사용자가 "뭘 하는 앱인지" 즉시 파악 가능

2. **시간대별 그룹핑(아침/저녁)**
   - 여러 약을 복용하는 박정호 같은 사용자가 한눈에 파악 가능

3. **달력 색상 시각화(초록/노랑/빨강)**
   - 복약 이력을 패턴으로 이해 가능

4. **체크 후 카드 배경 변화**
   - 시각적 피드백이 명확해서 "됐다"는 느낌 제공

### 심각한 결함

1. **데이터 영속성 없음**
   - MVP 단계의 완전한 한계
   - 박정호 같은 일반 사용자가 바로 이탈하는 이유

2. **푸시 알림 미구현**
   - 기획서 명시 기능인데 코드에 전혀 없음
   - "바쁜 자영업자" 페르소나에게 가장 필요한 기능

3. **미복용 약 인앱 알림 없음**
   - "어제 N개 약을 놓쳤어요" 배너가 없음
   - 유저가 스스로 달력을 찾아야 함

4. **글씨 크기 및 터치 영역 부족**
   - checkMeta fontSize 12 (권장 14 이상)
   - checkButton 32×32 (권장 44pt)
   - 노안 초기 + 손가락 정밀도 낮은 사용자 배제

5. **연속 등록 흐름 부재**
   - 저장 후 바로 메인으로 이동
   - 기획서의 "다른 약도 등록할까요?" 미구현

### 중간 우선순위 개선

- 달력 접근성 개선 (스크롤 수 줄이기)
- 달력 범례 추가
- 약 편집 기능 (삭제가 아닌 수정)
- 식단 기록 기능 (기획서 명시)
- 기록 내보내기/공유 (병원 진료 대비)

## 최종 결론

박정호는 **GLP-1 약물 + 다중 복약 관리 유저의 전형**이다. 그는:

- 약을 챙기고 싶은 의지가 있고 (아내 권유도 있고)
- 병원 진료 때 기록을 보여주고 싶어하고
- 기본 UI는 직관적으로 이해할 수 있다

하지만 현재 GLP-Care는:

- 데이터가 저장되지 않아서 (치명적) 신뢰할 수 없고
- 저녁 약 빼먹을 때 알림이 없어서 (바쁜 유저 배제) 보호 기능 부족하며
- 글씨가 작고 터치 영역이 작아서 (노안 초기 사용자 배제) 접근성이 낮다

**NPS 4/10, 3개월 후 이탈률 85%는 이 결함들의 직접적 결과다.**

정호가 친구에게 추천하기 위해서는:
1. 데이터 저장 (필수)
2. 푸시 알림 (필수)
3. 접근성 개선 (높은 우선순위)

이 세 가지가 최소한 구현되어야 한다.
